# 风险模型

BinSight 把“观察到的证据”和“对风险的解释”分开处理。

## 风险等级

- `low`：异常但信号较弱。
- `medium`：存在可疑能力或安全加固不足。
- `high`：存在强风险能力，或多个可疑信号互相印证。
- `critical`：有直接危险行为证据，并且上下文支持较强。

风险等级不等于恶意意图。能力类规则即使是 `medium`，含义也应是“需要审查该能力”，
而不是“该文件就是恶意软件”。

## 风险类型、置信度和证据强度

每条规则命中都会包含：

- `risk_type`：`capability`、`suspicious` 或 `malicious-likely`。
- `confidence`：`low`、`medium` 或 `high`。
- `evidence_strength`：`weak`、`medium` 或 `strong`。

单个导入函数、单个 URL、模型提供方 URL、API key 环境变量名都属于弱证据。
高风险结论应来自强证据，或多个相互印证的独立信号。

## 证据类型

- 导入库。
- 导入函数或符号。
- 可疑字符串。
- 节区属性。
- 安全加固缺失。
- 反汇编片段。
- RAG 知识条目。

## 降低误报

规则应描述“能力”，不要直接推断“意图”。例如 `CreateRemoteThread` 表明远程线程能力，不等于一定是恶意软件。报告必须区分确定证据和推测风险。

默认规则尽量使用组合条件。例如，命令执行 API 单独出现只产生能力提示；
命令执行 API 与危险命令字符串同时出现时，才会升级为高风险。
网络 API 单独出现是低风险能力证据；网络 API 与内嵌目标地址同时出现才是可疑行为，
但仍不自动等同恶意。

## 最终评估融合

启用在线 AI Provider 时，BinSight 会保留本地基线和 AI 评估，再写出融合后的 `final_assessment`。

- 如果 AI 不可用或 JSON 解析失败，最终评估回退到本地分析。
- 本地存在 `high` 或 `critical` 且 `malicious-likely`、`strong` 的强证据时，最终结论保留高风险下限。
- 如果本地只有 capability 类型命中，AI 可在上下文支持时降低最终等级。
- 如果 AI 发现本地规则未覆盖的更高风险组合，可以提升最终等级。
